# 多 Agent 攻坚（Multi-Agent Breakthrough）

你（根 agent）面对的是一个**很难**的问题：单条思路极容易在某个隐蔽处卡死，且你无法预先知道哪条路能走通。这个 skill 的全部目的，是把"一个人苦思"换成"一组 agent 在受控的竞争与对抗中各自深挖，你负责统领、审计、重定向"。

为什么不直接自己上、也不直接平均分派？因为人（和单个 agent）会被最早出现的"漂亮化简"吸引，所有努力悄悄收敛到同一条其实等价于原问题的死路上。要真的攻下来，必须**同时**养着几条互不相容的路线，并让独立的对抗者去戳穿每一份"看起来成了"的草稿。

下面是所有阶段都要遵守的原则。它们不是死板的命令，而是一套对抗"集体走偏"的机制——理解背后的动机，你就能在没写到的情形下自己判断怎么做。

---

## 何时使用本 skill

本 skill 由**用户显式调用**：当用户决定要用"一组 agent 并行探索 + 对抗审查 + 根 agent 统筹"的方式去攻坚时，调用它获取完整打法。典型信号：

- 用户说"组织/派一组 agent 来攻克/并行攻关这个难题""用多 agent 方式做对抗审查""给我一个多路线编排方案"等。
- 问题天然开放、没有标准解法，或公开信息里大概率找不到现成答案。
- 问题涉及多个可能互不兼容的建模角度（代数 / 结构 / 流 / 组合 / 计算验证 / 嵌入 …）。
- 错误答案代价高：一个隐蔽假设、循环论证、边界情况被忽略，就会让人误以为赢了其实没赢。

**不要**对以下情况启用（那用不上这种重型编排，且误触发会白白派大量 agent）：

- 普通单步任务：重构函数、画张图、翻译、改 API 格式。
- 单一 agent 就能完成的检索、总结、规划、解释类请求（包括"给我讲讲某个猜想为什么难"）。
- 具体调试类请求（除非用户明确要求多 agent 对抗式排查）。

## 核心纪律（贯穿全程）

### 1. 开局就要有真正多样化的组合

第一轮就派出一组**思路实质不同**的 agent，而不是"N 个 agent 做同一种方法的变体"。要求它们各自从不同的建模角度切入，例如：

- 不同的形式化（把问题翻译成哪种语言：图论 / 代数结构 / 流与割 / 范畴 / 优化 / 逻辑）
- 不同的不变量（什么量在变换下守恒，能用来归纳或排除）
- 不同的归约（化简成哪个已知问题，且要明确归约是否真的降低了难度）
- 不同的归纳或分解（按什么结构递归、分块）
- 计算性 sanity check（用小规模实例验证某个关键直觉是否站得住）

为什么：多样性是唯一能防止"全体收敛到同一条漂亮但无效的化简"的保险。开局同质，后面全是浪费。

### 2. 早期保持独立，别剧透"当前最看好哪条"

绝大多数 agent 在头几轮**不应该**知道你目前偏向哪条路线。让它们独立长大，直到各自暴露出真实的长处和缺口，再考虑交叉授粉。

为什么：一旦你告诉所有人"A 路线看起来最 promising"，独立探索就死了，大家会顺着 A 往下补，结果 A 的隐藏死穴没人去戳。独立性能换来真正不同的失败模式——而失败模式的差异，正是你判断哪条路还活着的依据。

### 3. 维护一份"方法族登记册"

在根 agent 这里显式记录：每个 agent 实际在用的**核心思想**（按数学/技术想法归类，而不是按表面措辞）。如果很多 agent 悄悄收敛到同一个方法族，把其中一部分**主动重定向**到尚未充分探索的建模角度。

为什么：表面不同的说法往往骨子里是同一招。按"想法"而非"措辞"分组，你才能看出多样性是不是真的在流失。

### 4. 防止单一路线靠"漂亮"垄断

一条路线如果最终停在"只需再证一个与原猜想等价的引理"，它**并没有更靠近完成**——除非它给出了这个引理的、真正全新的证明机制。不要因为某条路化简优雅就一直给它加注。

为什么：优雅的化简最容易骗人。等价归约让人产生"快成了"的错觉，却把难度原封不动藏进那个"显然的"子引理里。

### 5. 路线卡在"定理级缺口"就标记为阻塞

当某条路线止步于一个**其本身难度堪比原问题**的缺失引理时，把它标记为 blocked。只有在有人提出了**机制上真正新颖**的 invariant、构造或证明手段时，才重新给它派 agent。

为什么：在"需要另一个大定理"的死路上反复投人是沉没成本。封锁 ≠ 放弃，而是把火力转移到还有新鲜血液的地方。

### 6. 让几条互不相容的路线多轮并存

不要第一轮失败就砍掉大多数路线。让 2–3 条彼此冲突的证明路线活过好几轮，等各自独立长到能暴露真实强弱后，再做交叉授粉（把一个路线的构造借给另一个路线用）。

为什么：真正能走通的路线，早期往往看起来最不优雅。过早淘汰，你会把唯一活路杀掉。

### 7. 全程使用对抗 agent 审计

每份候选证明/方案都必须交给**独立的对抗 agent** 审查，专门找这些毛病：

- 关键性质是否真的逐条满足（例如"每条边恰好出现两次"这类精确计数是否真成立）
- 伪装成合法对象的非法构造（如把重复边的闭合迹当成"循环"、把平行边的 2-圈漏掉）
- 边界与退化情况：空对象、不连通结构、割点、归约过程中是否意外引入被禁止的结构
- **循环论证**：是否悄悄用了一个与原命题等价的陈述来当已知结论
- 模糊的"显然 / 例行 / 显然可兼容"——这些都是红旗，要求给出具体论证

为什么：人在自己写的东西上会 blind spot。独立对抗者是唯一可靠的"你其实没证出来"的报警器。

### 8. 只收具体的东西，拒收空话

要求每个 agent 返回**具体产物**：引理陈述、显式构造、方程、或对某个子引理的反例。明确拒绝：

- 纯状态汇报（"进展顺利，正在推进"）
- 含糊的乐观（"这个方向很有希望"）
- 声称某个全局兼容性"是例行的"——例行也要给证明

为什么：没有具体产物的 agent 轮次，对攻坚零贡献，只会制造"我们在动"的假象。

### 9. 根 agent 的职责：综合、挑战、重定向、再发新一轮

你不是派完就等。每一轮结束后你要：

1. 读所有产物，按方法族登记册归并。
2. 找出哪条路线被对抗审查戳穿了、哪条还站着。
3. 挑战还站着的路线最薄弱的一环。
4. 把火力从 blocked 路线转移到新鲜角度，或重新激活某条路线（仅当有新机制）。
5. **发起新一轮**，而不是第一轮失败就收工。

不要在第一波失败后就停。只有某条路线**存活了完整对抗审计**才算数。

### 10. 不提前返回

不要因为"当前路线都失败了"或"agent 报了定理级缺口"就回来交差。继续发新一轮；只有出现**真正新颖的机制**才重开被封锁的路线；持续寻找全新建模角度。

返回条件只有两个：
- 拿到一份**完整的、通过对抗审计**的肯定性解答；或
- （若始终没有）只报告**最强、被严格证明的那段推导**，并精确说明它剩下的缺口是什么。

除此之外，不要返回：不要返回"一个化简"、不要返回"部分结果"、不要返回"孤立的缺失引理"、不要返回"尽力了"的总结、不要返回"这题很难"的解释。

为什么：攻坚的价值全在"真的拿下"。提前返回的每一次，都是在用一份半成品冒充成果。

### 11. 对外部搜索的约束

公开搜索**只能**用于查普通背景知识或标准命名定理，不能拿去搜"这个确切问题怎么解"或"这个问题是不是还没解决"。不要靠搜网页来判断问题是否开放，更不要因此就回答"它是开放的"。

为什么：本 skill 的前提是你要靠 agent 编排自己攻，而不是抄现成答案或提前认输。

## 如何实际派发 agent（实操）

这一节把上面的原则落到"你具体怎么发任务"。在本环境里，agent = 你用并发任务机制派出的子 agent；同一轮里尽量**一次性并发派出**多个，让它们真正并行、互不干扰。

### 派发探索 agent 的任务模板

给每个探索 agent 的任务里，要逼出具体产物、并固定它的"方法族"，但**不要把其他路线的结论塞给它**：

```
你是攻坚小组的一员。我们正在攻克这个问题：
<用一两句精确重述问题，含所有边界条件>

你的专属切入点（只做这一种，不要参考其他路线）：
<角度 A：例如"尝试用 X 不变量做结构归纳" / "翻译成流与割的语言" / "找小规模反例做计算 sanity check">

要求：
- 返回具体产物：引理陈述 + 证明草稿，或显式构造，或针对某子引理的反例。
- 如果止步于一个"还需再证 Y"的缺口，明确写出 Y 的陈述，并判断 Y 是否与原问题难度相当。
- 不要写状态汇报；不要说"很有希望"而不给论证。
- 不要搜索该问题的现成解答；只能查普通背景知识。
```

注意：给不同 agent 的"切入点"要来自 §1 列出的**不同方法族**；给 agent 的指令里不要出现"目前最看好路线是 B"这种剧透。

### 派发对抗审查 agent 的任务模板

对每份通过初筛的候选，单独派一个**没参与该路线**的对抗 agent，专门按 §7 找茬：

```
请作为独立审查者，审计下面这份候选方案。你的唯一目标是证伪或挑出漏洞，不要帮它圆。
<候选方案全文>

逐项检查并给出证据：
1. 关键精确性质是否真成立（逐条，含计数/边界）。
2. 是否存在"伪装成合法对象的非法构造"。
3. 边界与退化情况是否覆盖（空、不连通、割点、归约是否引入被禁结构）。
4. 是否存在循环论证（是否偷偷用了与原命题等价的陈述）。
5. 是否有"显然/例行"这类未被证明的跳跃。
结论：通过 / 存在致命漏洞（列出），并说明漏洞是否可修。
```

### 方法族登记册（根 agent 维护）

每轮用一张表记录，便于发现"多样性流失"：

```
| 路线 ID | 方法族（核心思想） | 当前状态(open/blocked/stand) | 本轮产物摘要 | 对抗审计结论 |
|--------|------------------|------------------------------|--------------|--------------|
| R1     | 结构归纳 + 不变量X | stand                        | 证到引理 L   | 边界 case 缺失 |
| R2     | 流与割归约        | blocked                      | 停在等价引理 | 等价原问题    |
```

状态含义：
- **open**：还在推进，且无定理级缺口。
- **stand**：通过对抗审计、当前无致命漏洞。
- **blocked**：卡在定理级缺口；除非出现新机制，否则不再加注。

当多个路线悄悄落到同一方法族时（看第二列），把其中一两个重定向到表里空白的角度。

## 一个典型的轮次循环（给根 agent 的操作清单）

1. **开局**：并发派出 6–12 个探索 agent，覆盖 §1 列出的多种不同角度，互不告知彼此（用上面的探索模板，每个换一个切入点）。
2. **收件 + 登记**：读所有产物，按方法族填入登记册，剔除空话（§8）。
3. **对抗审计**：对每份像样的候选，并发派独立对抗 agent 按 §7、用对抗模板找茬。
4. **决策**：据登记册与审计，决定哪些 blocked、哪些重定向、哪些加注（§4–§6）。
5. **再发一轮**：带上上轮结论给需要协调的 agent，但**不要**剧透给应保持独立的探索 agent，启动下一轮。
6. 重复直到满足 §10 的返回条件。

## 返回格式

若攻下：

```
## 完整证明 / 方案
[核心思路一句话]
[按方法族组织的完整推导，每条关键性质标注已通过对抗审计]

## 对抗审计结论
[每条路线被戳穿的点 / 最终存活路线的审查结果]
```

若未攻下：

```
## 已被严格证明的最强推导
[具体到了哪一步，用了什么构造/引理]

## 精确剩余缺口
[卡在哪个定理级引理，为什么它等价于原问题或仍需要新机制]
```
