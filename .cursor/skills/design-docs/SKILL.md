# 实现级设计文档编写规范

## 文档定位

设计文档是**开发合同**，不是讨论材料。写完即可直接编码，不需要二次解读。

## 设计流程

按以下阶段推进，每个阶段产出明确产物，在用户确认后再进入下一阶段。

### 阶段一：需求分析

**产物**：对问题域的完整理解。

- 明确项目的核心目标和边界——做什么、不做什么
- 列出所有功能模块和它们的行为规格
- 识别关键技术约束（分发方式、启动速度、跨平台、依赖限制等）
- 如果是重写/迁移项目：逐函数分析原实现，确保无遗漏

### 阶段二：技术选型

**产物**：技术栈决策表。

用多维对比表评估候选方案，维度至少包括：

| 维度 | 方案 A | 方案 B |
|------|--------|--------|
| 核心需求匹配度 | ... | ... |
| 开发效率 | ... | ... |
| 运行时性能 | ... | ... |
| 分发/部署复杂度 | ... | ... |
| 依赖数量与维护成本 | ... | ... |
| 生态成熟度 | ... | ... |

**关键框架/库的选型必须做影响分析**：不只比较功能，还要分析采用后对架构的约束。例如：
- 框架是否接管了某些资源（I/O、线程、生命周期）？能否配置绕过？
- 框架的测试支持是否覆盖项目的测试模式？
- 引入的传递依赖有多少？包体大小增加多少？
- 框架的行为细节是否影响已有测试的断言？

选型结果必须给出明确结论，不留"A 或 B 都行"的模糊空间。

### 阶段三：文档编写

**产物**：按模块拆分的设计文档集。

详见下方「文档组织」和「内容标准」。

### 阶段四：审查与修正

**产物**：经过交叉验证的最终文档。

详见下方「审核清单」。

## 文档组织

### 按模块拆分为独立文件

每个文件聚焦一个子系统，**只关心自身的实现细节**，不依赖其他模块的内部实现：

```
docs/
  architecture.md    # 总体架构（项目结构、模块关系、数据流）
  auth.md            # 认证模块（登录、会话、权限）
  storage.md         # 存储层（数据访问、缓存策略）
  ...
```

文件名和拆分粒度根据项目实际情况确定，原则是**一个文件对应一个可独立实现和测试的子系统**。

### 自包含原则

> **每份文档必须包含完成该模块实现所需的全部信息。**

读者只看一份文档就能实现对应模块，不需要翻阅其他文档理解上下文。具体要求：

- 模块需要的外部输入，在文档开头以类型定义的形式声明（只定义结构，不描述来源模块的实现）
- 模块的输出接口，在文档末尾完整定义
- 不引用其他模块的内部实现细节（如"参见 xxx.md 中的 YYY 函数"）

### 接口归属生产方

**接口定义放在产出方的文档中**，消费方只声明"我需要什么结构"。

理由：修改接口时只需改一处。如果接口定义散落在消费方文档中，一次变更需要同步多份文件。

### 总体架构文档

`architecture.md` 是唯一允许跨模块描述的文件，包含：

- 项目定位与核心设计决策
- 技术栈声明
- 目录结构
- **模块关系图**（文本形式，展示数据流向）
- 环境变量/配置表
- 全局共享的工具函数

### 跨文档引用

不重复，只引用。需要时用一行指向权威位置：

```markdown
认证令牌刷新策略详见 `auth.md`。
```

## 架构健壮性

> **好的设计让后续修改只涉及最少的文件。**

### 模块边界清晰

每个模块通过明确的输入/输出类型与外界交互，内部实现对外不可见。模块间不共享内部类型——如果两个模块需要同一种数据，定义各自的类型并在边界处转换。

### 变更影响最小化

设计时对每种预期变更做推演，确保影响范围可控：

| 变更类型 | 理想影响范围 |
|----------|-------------|
| 新增一种业务操作 | 业务逻辑模块 + 路由/分派层的一个分支 |
| 修改展示样式 | 仅视图/渲染模块 |
| 替换底层存储 | 仅数据访问层 |
| 调整算法逻辑 | 仅算法模块 |

如果一个功能变更需要改三个以上模块，说明模块边界划分有问题，需要重新设计。

### 扩展点预留

在可预见的扩展方向预留接口，但**不过度设计**。只在以下情况预留：

- 已知近期会增加的功能
- 使用枚举/接口可以零成本预留的场景

不预留"也许将来有用"的抽象。

## 格式约束

| 约束项 | 限制 | 理由 |
|--------|------|------|
| 单文件行数 | ≤ 400 行 | 超过 400 行说明模块拆分不够细，应进一步拆分 |
| 代码示例 | ≤ 30 行/块 | 示例展示关键逻辑即可，完整实现不属于设计文档 |
| 标题层级 | 最深 `###`（h3） | 超过三级说明结构嵌套过深，应拆分为独立章节或文件 |
| 表格列数 | ≤ 6 列 | 超过 6 列改用类型定义或列表 |
| 模块文件数 | 一个子系统一个文件 | 不按"类型定义.md + 接口.md + 实现.md"拆分同一模块 |

## 内容标准

### 类型定义必须完整

每个数据类型列出所有字段、类型、用途：

```go
type Order struct {
    ID        string    // 订单唯一标识
    UserID    string    // 下单用户 ID
    Items     []Item    // 商品列表
    Status    Status    // 当前状态（枚举）
    CreatedAt time.Time // 创建时间
}
```

### 接口必须明确输入输出

函数签名 + 参数含义 + 返回值语义，不写"大概是这样"：

```go
// Authenticate 验证凭证，成功返回用户信息和令牌，失败返回错误。
func Authenticate(cred Credential) (*AuthResult, error)
```

### 优先级链必须编号

配置、命令分派等有优先级的逻辑用编号列表，最高在前：

```
1. 命令行参数（最高优先）
2. 环境变量
3. 配置文件
4. 硬编码默认值
```

### 状态用表格

模型状态、环境变量、命令映射等结构化信息用表格，不用散文。

### 组件间共享状态必须说明同步策略

当多个组件引用同一可变状态时，必须明确同步机制。例如：

```markdown
## sharedCache 同步

Controller 和 Renderer 共享同一个 map 引用。
**约束**：清除缓存时用 clear(cache)，
不要用 cache = make(map[string]Result)（会断开引用）。
```

没有同步说明的共享状态，实现时大概率出 bug。

### 模块间类型映射必须文档化

当两个模块定义各自的类型且需要互转时，写出完整转换代码：

```go
// service.User → api.UserResponse
resp := api.UserResponse{
    ID:   u.ID,
    Name: u.DisplayName,
    Role: mapRole(u.Permissions), // 权限集合 → 角色枚举
}
```

不写转换代码 → 两边类型各自完整也没用——**衔接处是最容易出错的地方**。

### 平台限制必须标注

涉及系统调用或平台相关行为的操作，标注已知限制和回退策略：

```markdown
`os.Rename` 不支持跨文件系统。source 和 dest 在不同挂载点时返回 EXDEV。
回退策略：捕获 EXDEV 后走 copy + removeAll。
```

## 语言规范

### 确定性语气

文档是最终规格，不是建议：

- ✅ "预计算查找表避免热路径浮点运算"
- ❌ "建议预计算查找表"
- ✅ "不执行操作"
- ❌ "不执行操作（或在 stderr 提示用户）"

### 禁止版本对比描述

文档只描述当前实现方案，不提及旧版本行为或版本间差异。实现时只会使用确定的一种方式，"v2 中不再…而是…"这类对比没有意义：

- ❌ "v2 中 Alt screen 不再通过 Program 选项设置，而是在 View() 中声明式启用"
- ✅ "Alt screen 在 View() 返回的 tea.View 中声明式启用"
- ❌ "样式不再直接调用 style.Render(text)，而是通过 Writer 控制"
- ✅ "样式通过 Writer 控制输出"

同理，"不再维护 X"、"不再硬编码 Y" 等暗示旧行为的表述，改为直接陈述当前设计。

### "可能"仅用于运行时事实

唯一允许使用"可能"的场景是描述运行时的不确定性（如"环境变量可能未设置"），不用于设计决策。

### 中文注释

代码示例中的注释使用中文，与实现保持一致。

## 第三方库 API 验证

文档中引用的第三方 API 必须与目标版本匹配。常见的版本间差异：

- 方法签名变化（参数增减、返回类型改变）
- 类型/方法重命名
- 架构模式变化（如选项式 → 声明式）

**处理方式**：对于无法在文档编写阶段 100% 确认的 API 细节，用 blockquote 标注：

```markdown
> 实现时验证：v2 中此方法的签名是否有变化。
```

标注过的不确定项在实现阶段逐一验证后移除。

## 设计变更协议

设计迭代中最常见的 bug 来源是**局部变更后遗漏了其他文档的同步更新**。

### 架构方案切换

当方案发生根本性变化时：

1. 先更新方案的权威文档
2. **全文搜索旧方案关键词**，逐一更新或删除
3. 重点检查：架构图描述、模块关系说明、函数命名前缀、代码示例、测试断言
4. 确认无残留后提交

### 概念重命名

1. 全文搜索旧名称的所有变体（函数名、类型名、文件名、UI 文案、注释、环境变量）
2. 逐一替换，不要批量盲替（同一概念可能出现为不同形式）
3. 检查对外接口：API 路径、环境变量名、配置文件键名
4. 更新测试文件名和测试描述

### 功能移除

残留最容易出现在：类型定义中的字段、注释中的枚举列表、加载逻辑中的特殊分支、测试用例和测试数据。

执行步骤：搜索功能相关的所有标识符和关键词 → 逐一删除 → 确认编译/逻辑完整性。

## 审核清单

编写完成后执行四轮审核：

### 第一轮：完整性

- 所有功能模块是否都有对应文档
- 每个模块的所有函数/类型是否都已记录
- 模块间的数据流是否完整（字段从入口定义 → 中间传递 → 最终消费，链条不断）

### 第二轮：跨文档一致性

- 同一函数/类型在所有出现位置的命名、签名、字段是否一致
- 方案是否统一（不能一处说"直接执行"，另一处说"生成脚本"）
- 函数参数是否匹配调用处传入的实际数据

### 第三轮：API 与平台正确性

- 第三方库 API 是否匹配目标版本
- 结构体字段是否与实际 API 一致
- 错误处理路径是否完整
- 系统调用是否标注了平台限制

### 第四轮：语言与变更残留

- 是否有"建议"、"也许"、"可以考虑"等不确定表述
- 是否有未决策的"A 或 B"选项
- 已切换的方案是否在其他文档中仍有旧描述
- 已移除的功能是否在注释、枚举、示例中仍有痕迹
