# 新功能实现流程

## 流程概览

```
1. 更新设计文档 → 2. 讨论确认 → 3. 编写测试 → 4. 实现功能 → 5. 验证并提交
```

每个阶段产出明确产物，用户确认后再进入下一阶段。

## 阶段一：更新设计文档

在动手写代码之前，先更新 `spec/` 目录下的相关规格文档。

### 操作步骤

1. 阅读现有相关文档（用 Read 工具），理解当前设计
2. 确定本次变更涉及哪些文档需要更新
3. 修改文档，内容包括：
   - 新增的类型定义和接口签名
   - 配置项及其优先级链
   - 行为规格（如渲染逻辑、状态转换）
4. 文档遵循 `design-docs` skill 中的规范（确定性语气、自包含、无版本对比）

### 产物

已更新的设计文档，可直接用于指导实现。

## 阶段二：讨论目标

将设计方案总结呈现给用户，等待确认。

### 操作步骤

1. 用简洁的文字总结变更方案（解决什么问题、技术方案、影响范围）
2. 使用 AskQuestion 工具请用户确认或提出修改意见
3. 如果用户有修改意见，回到阶段一调整文档后重新确认

### 要求

- 不要在用户确认前开始写实现代码
- 方案必须具体到可以直接编码，不留模糊空间

## 阶段三：编写测试

先写测试（TDD），明确预期行为，测试此时应当编译失败或运行失败。

### 操作步骤

1. 在对应的 `_test.go` 文件中添加新测试用例
2. 测试应覆盖：
   - 正常路径（happy path）
   - 边界条件
   - 与现有功能的交互（确认不破坏旧行为）
3. 运行测试确认它们因功能未实现而失败（红色阶段）
4. 遵循 `tui-testing` skill 中的测试原则（测试功能而非代码、真实依赖）

### 要求

- 测试用例使用 data-driven 风格（表驱动测试）
- check 辅助函数封装断言逻辑，API 变更时只改一处
- 至少有一个测试走真实用户操作的代码路径

## 阶段四：实现功能

根据设计文档和测试用例编写实现代码。

### 操作步骤

1. 按模块逐个实现（每完成一个模块就运行测试确认）
2. 实现顺序建议：底层模块 → 上层集成
3. 关键注释使用中文
4. 每完成一个独立部分可单独提交

### 要求

- 不偏离设计文档的规格
- 新增代码不引入 linter 错误（用 ReadLints 检查）
- 保持与现有代码风格一致

## 阶段五：验证并提交

### 操作步骤

1. 运行全量测试：`go test ./... -count=1`
2. 确认所有测试通过（新测试 + 旧测试）
3. 检查 linter 错误
4. **文档同步检查**（不可跳过）：
   - 对照 `spec/` 目录下相关文档，确认本次变更涉及的函数签名、结构体、行为描述、配置项等是否需要更新
   - 对照 `README.md`，确认用户可见的功能说明是否需要更新
   - 文档更新与代码变更在同一个 commit 中提交
5. 提交 commit（遵循项目 commit 规范）

### 提交规范

- 每个完整功能一个 commit（设计文档 + 测试 + 实现一起）
- commit message 格式：`feat/fix/refactor: 简要描述`
- 依赖变更（go.mod/go.sum）一并纳入
- 文档更新一并纳入，不单独提交

## 异常处理

| 情况 | 处理方式 |
|------|----------|
| 测试通过但覆盖不足 | 补充边界条件测试后再提交 |
| 实现偏离设计文档 | 先更新文档使其一致，再继续 |
| 旧测试被破坏 | 分析原因——如果是预期的行为变更则更新测试，否则修复实现 |
| 编译错误 | 立即修复，不积累 |
| 文档与代码不一致 | 以代码为准更新文档，在同一 commit 中提交 |
